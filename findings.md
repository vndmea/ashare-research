# 发现与决策

## 需求
- 用户希望把当前项目的开发计划进一步细化，形成可以连续执行的阶段路线图。
- 用户进一步要求审查三个规划文件是否需要更新、细化和补充。
- 当前项目目标是 A 股日频量化研究平台，不是承诺收益的实盘系统。
- 用户希望后续开发能围绕现有架构继续推进，而不是重新推翻重做。

## 研究发现
- 当前项目已经完成一轮关键架构升级：配置类型化、统一研究流水线、策略注册表、回测模块拆分、实验扫描模块、归因模块、中文看板。
- 当前测试状态稳定，最近一次本地验证为 27 passed。
- 仓库仍处于大规模未提交状态，说明当前更适合继续按阶段推进，而不是立即做发布整理。
- 看板浏览器显示内容已经中文化，后续只需在新增功能上保持同样策略。
- README 中 Next Milestones 的方向与当前项目真实需求基本一致，但还不够细。
- task_plan.md 的主要缺口是“当前阶段未推进、缺阶段依赖、缺风险缓解、缺更具体的阶段完成定义”。
- findings.md 的主要缺口是“缺当前阻塞、缺下一步建议、缺对计划文件自身的审查结论”。
- progress.md 的主要缺口是“缺本次审查与补强动作记录”。
- 当前项目架构的短板主要不是分层，而是数据契约仍偏隐式、策略参数模型仍偏单策略、回测结果还缺执行诊断层。
- 如果要继续逼近优秀工具，最有效的追赶主线不是页面，而是：数据平台最小内核、交易账本与执行诊断、通用策略规格、实验复现体系。
- 阶段 2 的第一项已经开始落地：项目内已新增 `src/ashare_research/contracts/schemas.py` 作为官方数据契约真源。
- 当前 contracts 已覆盖 source datasets 与 runtime/report datasets，并记录字段、主键、语义、生产模块、消费模块。
- `src/ashare_research/data/daily_bars.py`、`benchmarks.py`、`calendar.py`、`universe.py` 已开始引用 contracts 层，说明数据治理正在从“文档约定”转向“代码约束”。

## 当前阻塞与判断
- 当前最实际的阻塞不是架构，而是真实数据链路和研究真实性还没有进入执行阶段。
- 当前规划文件已经可用，但如果不把阶段状态从阶段 1 推进到阶段 2，会让后续执行失去方向感。
- apply_patch 在当前桌面环境下存在权限问题，因此文件维护需要明确记录这个限制。
- 当前真正需要预防的是后续开发过程中出现“平行主流程、平行配置、平行状态、平行指标计算”。
- 当前如果马上大规模扩策略和实验，最先出问题的会是字段语义漂移和配置模型耦合，而不是模块不够多。
- 当前下一步最值得继续收紧的点，不是再列更多结构，而是让更多校验逻辑和报表输出直接依赖 contracts，而不是依赖隐式列名假设。

## 技术决策
| 决策 | 理由 |
|------|------|
| 采用阶段化路线图管理后续开发 | 当前任务已明显超过一次性小改范围，需要持续跟踪 |
| 先做真实数据和回测真实性，再扩展策略 | 没有真实数据和可信撮合，策略扩展的价值有限 |
| 继续使用展示层中文映射而不改底层字段 | 保持报表结构稳定，避免影响测试与研究代码 |
| 维持研究优先定位 | 当前项目离实盘还有明显距离，产品边界需要清楚 |
| 先补强规划文件，再进入阶段 2 开发 | 让后续执行有明确依赖、风险和验收边界 |
| 用统一主流程、统一配置真源和统一状态对象约束后续开发 | 防止模块各自为战、功能重复和数据不互通 |
| 在规划文件中显式补入“数据契约与领域模型治理” | 长期扩展时，真正的脆弱点是隐式 DataFrame 契约 |
| 将“差距缩小专项计划”并入 task_plan.md | 让后续开发不只追求可用，还能持续逼近优秀工具能力边界 |

## 遇到的问题
| 问题 | 解决方案 |
|------|---------|
| 仓库中没有现成的计划文件 | 新建 task_plan.md、findings.md、progress.md 作为持久规划文件 |
| 当前计划容易流于泛化 | 结合仓库真实模块和现有未提交变更重新组织为可执行阶段 |
| 规划文件初版偏静态 | 增加当前阶段、依赖、风险、下一步建议和本次审查日志 |
| apply_patch 临时执行链路不可用 | 本轮先用本地文件写入兜底，并记录为环境限制 |
| 后续开发可能出现平行实现 | 在 task_plan.md 中增加架构守则、开发红线、模块协作约束和执行检查清单 |
| 数据结构长期可能变松散 | 在 task_plan.md 中增加官方结构清单、治理要求和领域对象演进方向 |
| 后续开发可能只补表面功能 | 在 task_plan.md 中增加差距缩小专项计划，约束优先级聚焦核心能力 |

## 下一步建议
1. 按 task_plan.md 正式从阶段 2 开始执行。
2. 在现有 `contracts/schemas.py` 基础上继续补“校验器复用”与“最小真实数据样例”，让 contracts 不只是清单。
3. 在进入阶段 3 前，先把当前回测假设做成显式清单。
4. 后续每次进入新阶段时，都同步更新 progress.md 和 findings.md。
5. 每次新增功能前先过一遍“执行检查清单”，防止出现第二条主流程或第二套状态。
6. 阶段 2 后续重点应转向字段错误、日期错位、缺失值和复权链路验证，不再停留在静态结构定义。
7. 后续优先推进差距缩小专项计划中的前四条主线，而不是优先美化页面。

## 架构防偏结论
- 当前项目可以通过一条统一数据流来防偏：`raw data -> standardized bars -> signals -> backtest result -> reports -> dashboard`。
- `run.py` 应长期作为唯一主链；任何新能力都应挂在这条链上，而不是平行生长。
- `ResearchConfig` 应长期作为唯一配置真源；不能让实验、看板、策略各自维护私有配置。
- 共享状态应统一通过 `ResearchArtifacts`、`ResearchRun`、`ResearchRunWithReports` 这类结构化对象传递。
- 看板、CLI、实验模块必须消费统一结果，而不是各自重算一遍核心指标。
- 当前项目属于“流程架构强于数据架构”的状态，因此下一步必须优先补强数据契约，而不是继续无节制扩功能。
