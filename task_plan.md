## 风险与缓解
| 风险 | 影响 | 缓解方式 |
|------|------|---------|
| 真实数据字段不完整 | 阶段 2 卡住 | 先定义最小可运行字段集，再逐步加字段 |
| 回测真实性增强过多导致复杂度失控 | 阶段 3 延期 | 先明确 MVP 级约束，再分批实现 |
| 策略扩展过早开始 | 研究结果不可信 | 保持“先数据、后回测、再策略”的顺序 |
| 看板展示先行 | 容易做出表面功能 | 强制让看板跟随研究输出成熟度推进 |
| 仓库未提交改动较多 | 后续回滚和审查困难 | 每完成一个阶段就整理一次提交边界 |
| DataFrame 字段语义漂移 | 模块间结果不一致 | 先定义统一契约，再允许扩字段 |
| 策略参数模型耦合到单一策略 | 策略扩展时牵一发动全身 | 先抽象通用参数模型，再接新策略 |
| 回测结果缺少执行诊断层 | 后续模块各自维护状态 | 在阶段 3 统一引入交易/执行中间结果模型 |

## 数据契约与领域模型治理
### 当前判断
1. 当前主流程架构已经基本成型，但数据结构仍偏向“DataFrame 约定俗成”，长期会有漂移风险。
2. 现在真正需要提前治理的不是页面，而是字段语义、官方结构、共享状态和中间结果模型。
3. 这部分必须在阶段 2 和阶段 3 期间完成第一版，不适合拖到后期。

### 官方结构清单
当前已在 `src/ashare_research/contracts/schemas.py` 中落地第一版官方 contracts/schemas，并由 `tests/test_contracts.py` 覆盖基础一致性校验。

1. `bars`
2. `benchmark_returns`
3. `signals`
4. `positions`
5. `equity_curve`
6. `drawdowns`
7. `rolling_metrics`
8. `monthly_returns`
9. `industry_exposure`
10. `strategy_attribution`

### 治理要求
1. 每个官方结构都要定义：必填列、可选列、字段类型、字段语义、生成模块、消费模块。
2. 新字段进入主流程前，必须先补到统一契约说明，而不是只在某个函数里直接使用。
3. 同一语义只允许一个官方字段名，禁止同义不同名长期并存。
4. 报表层可以做展示映射，但不能创造新的核心业务语义。
5. 若某个结构已经承担了跨模块共享职责，优先补契约，不允许继续“边用边猜”。
6. 输入源 loader 的必填字段校验应直接复用 contracts 中的官方定义，避免 loader 与文档各维护一份字段真源。

### 领域对象演进方向
1. 继续保留 `ResearchArtifacts`、`ResearchRun`、`ResearchRunWithReports` 作为主流程壳对象。
2. 中期应考虑补充显式领域对象或至少显式 schema：
   `SignalFrame`
   `PositionFrame`
   `TradeLedger`
   `ExecutionDiagnostics`
3. 在阶段 3 之前，不强行全面对象化；但要先把哪些结构将来需要升级明确写清楚。

### 完成定义
1. 项目内出现的核心 DataFrame 都能在统一清单里找到定义。
2. 新模块开发时不再需要通过读源码猜列名和语义。
3. 看板、实验、回测、分析共用同一套字段口径。
4. 当引入第二类以上策略时，不需要反复修改核心配置骨架来适配单一策略。

## 差距缩小专项计划
### 目标

3. 缩小差距的关键不是堆更多页面，而是先补核心能力底座。

## 架构守则
1. `src/ashare_research/pipeline/run.py` 是唯一研究主流程入口。
2. 所有新参数必须先进入 `src/ashare_research/config.py` 的 `ResearchConfig`，禁止模块私有配置真源。
3. 展示层不得重算核心研究结果；`dashboard.py` 只消费统一报表或统一运行结果。
4. `data/` 只负责加载、校验、标准化；`strategies/` 只负责信号；`backtest/` 只负责撮合与持仓演化；`analysis/` 只负责指标、归因、报表。
5. 同一领域概念只能有一套官方结构和命名，例如 bars、signals、positions、equity_curve、reports。
6. 所有共享状态必须通过结构化对象传递，优先复用 `ResearchArtifacts`、`ResearchRun`、`ResearchRunWithReports`，禁止散落的全局状态和临时字典真源。
7. 新功能先回答“复用谁”，再考虑“新建什么”；若能扩展现有模块，不允许平行造轮子。

## 开发红线
1. 不允许新策略直接绕过 `run.py` 自行读取数据并产出独立结果链路。
2. 不允许新看板、新 CLI、新实验模块各自维护一套指标计算逻辑。
3. 不允许同一含义出现多套字段名，例如 `rebalance_frequency` 与 `rebalance_interval` 并存。
4. 不允许在不同模块中各自维护“当前状态”副本，导致状态漂移。
5. 不允许未经过字段契约确认的数据直接进入回测与报表。

## 模块协作约束
| 模块 | 允许输入 | 允许输出 | 禁止行为 |
|------|---------|---------|---------|
| `data/` | 原始 CSV、配置 | 标准化数据表 | 直接生成策略结论或报表 |
| `strategies/` | 标准化 bars、策略参数 | signals | 自行加载原始数据、自行写回测结果 |
| `backtest/` | bars、signals、约束、基准 | `BacktestResult` | 自行定义平行报表格式 |
| `analysis/` | `BacktestResult`、bars、benchmark | 报表与诊断表 | 改写交易逻辑 |
| `dashboard.py` | 统一报表文件、配置预览 | 浏览器展示 | 重算核心收益、重算回测状态 |
| `experiments/` | 统一配置与主流程入口 | 批量实验结果 | 绕过统一流水线 |

## 执行检查清单
1. 新功能是否接入了统一主流程 `run.py`。
2. 新参数是否已经进入 `ResearchConfig`。
3. 新增字段是否写入字段契约并有校验。
4. 新输出是否复用统一报表链路。
5. 是否复用了现有模块，而不是新建平行实现。
6. 是否引入了新的重复状态、重复命名或重复计算。
7. 是否补上跨模块测试与看板/CLI 兼容验证。
8. 新增或变更字段是否已经更新数据契约说明。
9. 新功能是否引入了新的中间状态对象；若引入，是否已经明确其官方地位和消费边界。
10. 新能力是否在“差距缩小专项计划”的主线中，还是只是表面功能堆叠。

## 验收标准
1. 使用真实 A 股数据时，项目可稳定完成加载、回测、报表导出和看板展示。
2. 回测结果能解释主要假设，且关键撮合与交易约束有测试覆盖。
3. 新增策略可通过注册表接入，并支持配置化运行。
4. 参数扫描结果可被批量汇总和可视化比较。
5. 浏览器可见内容保持中文，研究输出口径一致。
6. 每个阶段完成后都能明确“完成定义”和下一阶段输入。

## 关键问题
1. 第一批真实数据源最终采用哪一套供应方式，是继续公共下载，还是接入更稳定的数据供应链。
2. 回测真实性的第一优先是滑点模型、成交约束还是成分股历史还原。
3. 策略扩展优先做横截面选股，还是先把现有趋势类策略做成模板。
4. 真实数据样例是否需要单独建立一份轻量测试夹具，避免仓库过重。

## 已做决策
| 决策 | 理由 |
|------|------|
| 先完成架构再补研究能力 | 当前架构已具备继续扩展的基础，不需要再次大拆 |
| 浏览器展示层统一中文化 | 满足当前产品化要求，同时不破坏底层数据结构 |
| 当前阶段优先真实数据与回测真实性 | 这是让项目从样例走向可用研究平台的关键路径 |
| 保持研究优先而非实盘承诺 | 当前项目定位是研究与验证，不做收益承诺 |
| 当前阶段推进到阶段 2 | 阶段 1 的计划落盘已经完成，应该正式进入执行态 |
| 当前阶段同步补入数据契约与领域模型治理 | 长期扩展时，真正脆弱的是隐式 DataFrame 契约 |
